## 追加记录：Phase 1.1 `need_reobserve` 边界稳定化

### 本次目标

真实在线摄像头验证已经证明：

- `need_reobserve` 可以在真实 ROS2 在线链路中触发

- 不改模型、不训练、不扩数据集

### 稳定化规则

当前 raw `need_reobserve` 候选仍来自：

- `low_visibility`
- `occluded`
- `window_not_ready`

supervisor 现在增加最小滞回：

- 连续 `reobserve_enter_frames=2` 帧 raw 候选才进入 `need_reobserve`
- 连续 `reobserve_exit_frames=5` 帧非 raw 候选才退出回 `monitor`
- `fall_detected`、`no_person_present`、`unavailable/error/completed` 仍立即按原优先级处理，并重置 reobserve 滞回计数

当前状态优先级仍保持：

1. `fall_detected`
2. `no_person_present`
3. `need_reobserve`
4. `monitor`
5. `hold / degraded`

### 最小验证案例更新

`scripts/verify_need_reobserve_case.py` 已新增：

- `reobserve_hysteresis_jitter_case`

该案例验证：

- 单帧 `low_visibility` 不会立刻进入 `need_reobserve`
- 连续 2 帧 `low_visibility` 才进入 `need_reobserve`
- 已进入 `need_reobserve` 后，单帧 `stable` 不会立刻退出
- 连续 5 帧 `stable` 才退出回 `monitor`

### 未解决风险

- 当前滞回以帧数为单位，不是时间窗口；不同摄像头帧率下体感延迟会不同。
- 默认 `2 / 5` 是保守工程默认值，仍需真实在线观察后再决定是否调成 `3 / 8` 或其他组合。
- `low_visibility` 与 `occluded` 的 raw 判据仍来自现有关键点诊断字段，不是真值遮挡标签。

### `TaskPlannerStatus` 字段收口

当前 status 会明确区分：

- `reason`：当前实际状态原因
- `state_reason`：由动作映射得到的占位状态原因
- `request_reason`：原始请求原因
- `request_supported`：占位 planner 是否支持该动作

对于不支持的 `requested_action`：

- `planner_state=invalid_request`
- `reason=unsupported_requested_action`
- `request_supported=false`

这样可以避免上游自由文本 reason 掩盖 unsupported action。

### 最小接入方式

外部依赖：

```bash
sudo apt install ros-humble-turtlebot4-simulator ros-humble-irobot-create-nodes ros-humble-rtabmap-ros
```

启动入口：

```bash
cd ros2_ws
source /opt/ros/humble/setup.bash
colcon build --symlink-install --packages-select yolopose_ros
source install/setup.bash
ros2 launch yolopose_ros phase4a_turtlebot4_rtabmap.launch.py
```

默认逻辑 topic：

- RGB：`/oakd/rgb/preview/image_raw`
- Depth：`/oakd/rgb/preview/depth`
- Camera info：`/oakd/rgb/preview/camera_info`
- Scan：`/scan`
- Odom：`/odom`
- TF：`/tf`、`/tf_static`

当前默认禁用 `rgbd_odometry`：

- `visual_odometry=false`
- `publish_tf_odom=false`

这样 `/odom` 只保留 TurtleBot4 底盘侧 `diffdrive_controller` 一个 publisher，避免 RTAB-Map visual odometry 与底盘里程计同时发布 `/odom` 导致 RViz 位姿跳变和地图重影。

如 TurtleBot4 仿真实际 topic 不同，仍可通过 launch 参数覆盖 `rgb_topic / depth_topic / camera_info_topic / scan_topic / odom_topic`。

### 结构性结论

- Phase 4a 已从“文档中的 RTAB-Map 挂载边界”推进为“可启动的最小集成入口”。
- 当前接入方式仍是空间层 sidecar，不改变感知、监督和 planner placeholder 的冻结语义。
- `RTAB-Map` 输出已经被明确为未来真实 planner 的空间状态来源，而不是当前 placeholder 的输入。
- 当前收口判据是：`/rgbd_odometry` 不启动，`/odom` 只有 `diffdrive_controller` 一个 publisher，`/map`、`/localization_pose`、`/tf` 可观察。

### 当前明确不做

- 不接真实 `Nav2` 完整任务闭环。
- 不接真实 `PlanSys2 / LTL`。
- 不做大规模家居仿真环境。
- 不新增自定义消息体系。
- 不做 Kalibr 标定流程。
- 不让 planner placeholder 消费 `/map` 或 `/localization_pose`。
- 不改模型、不训练、不扩外部数据集。

### 未解决风险

- 当前机器需要已安装 TurtleBot4 simulator 与 `rtabmap_launch`。
- 如果 TurtleBot4 仿真模型或版本变化，RGB / depth / camera_info topic 可能仍需重新 inventory。
- RTAB-Map 对 RGB-D 同步、`frame_id`、`odom` 和 TF 连通性敏感，第一轮只要求最小输出可见，不承诺地图质量。
