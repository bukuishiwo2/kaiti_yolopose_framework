## 追加记录：sequence 在线不出真的诊断增强

### 本次边界

- 不改默认 `LSTM` 主线
- 不改模型、不重训、不改权重
- 不回退到规则法主导
- 只增强 sequence 在线链路的可观测性

### 已完成

已补齐 sequence 在线调试字段，当前 `PerceptionEvent` 中可直接观察：

- `seq_model_loaded`
- `seq_detector_enabled`
- `seq_sequence_len`
- `seq_window_ready`
- `seq_window_size`
- `seq_track_id`
- `seq_fall_score`
- `seq_raw_fall_detected`
- `seq_stable_fall_detected`
- `seq_skip_reason`
- `seq_invalid_reason`

同时在 `/perception/debug_image` 叠加：

- `loaded / ready / win / mode`
- `track / valid / kpts / skip / invalid`

### 当前代码层结论

1. sequence checkpoint 在当前环境中可以成功加载。
2. 默认推理配置当前仍是 `mode=predict`，因此在线默认不会提供连续 `track_id`；
   sequence 分支会退回到“global primary-person sequence”路径，而不是 per-track 路径。
3. 因此，在线 sequence 不出真时，当前最需要先区分的是：
   - 窗口是否形成
   - 人体关键点是否有效
   - 实际分数是否长期远低于阈值
   而不是先去改阈值或重训。

### 当前最可能原因分类

在完成这轮诊断增强之前，最可能原因排序为：

1. `seq_window_ready` 未形成
2. 输入人体候选存在，但 `visible keypoints` 太少，导致特征质量差
3. 分数长期偏低，属于在线域偏移/动作风格与训练分布不一致
4. `track_id` 缺失导致当前只能走 global 模式，但这更像诊断线索，不应作为第一结论

### 统计结果

对整段 `5585` 帧事件做统计后得到：

- `seq_model_loaded_all = True`
- `seq_window_ready_frames = 5554`
- `seq_raw_true_frames = 812`
- `seq_stable_true_frames = 860`
- `rule_raw_true_frames = 2389`
- `rule_stable_true_frames = 2612`

按稳定触发片段统计：

- 规则法稳定触发段：`54`
- sequence 稳定触发段：`24`
- 规则法触发段中与 sequence 有重叠的：`20`
- 仅规则法触发、sequence 未覆盖的：`34`

### 当前结论

1. sequence 分支不是“没有工作”，而是“能触发一部分，但明显比规则法保守”。
2. 当前问题不属于：
   - 模型未加载
   - 窗口未形成
   - 关键点整体失效
3. 当前更像是：
   - 这类视频的时序分布与 `UR Fall` 主训练分布存在明显偏移
   - 拼接式参考视频对 `32` 帧窗口不友好
   - 默认 `mode=predict` 下 sequence 仍走 `global` 路径，而不是更稳定的 `track` 路径

### 工程判断

- 当前默认 `LSTM` 主线在 `UR Fall` 主 benchmark 上仍成立，不应因为单条外部参考视频而直接推翻。
- 但该视频已经说明：当前主线对“慢跌倒 / 家居风格 / 剪辑式外部视频”的泛化仍不够。
- 后续改进优先级不应是先调阈值，而应是：
  1. 先补家居/慢跌倒正样本分布
  2. 再做外部视频域验证
  3. 最后才考虑阈值或训练策略细化
