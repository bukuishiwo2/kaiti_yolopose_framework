# Kaiti Goal Alignment

本文件用于将当前仓库进展与课题文档 `kaiti.docx` 中的最终目标对齐，避免项目长期停留在“只做跌倒检测模型”的局部目标。

## 1. 开题文档中的最终目标

根据 `kaiti.docx`，最终课题目标是一个在 `ROS2` 框架下可运行、可评估、可在仿真与实物平台验证的移动机器人任务系统原型。系统需覆盖四层能力：

1. 动态环境自主建图与地图更新
- `Kalibr` 传感器标定
- `RTAB-Map` 在线建图与定位
- 动态环境下的增量更新与可达性支持

2. 目标识别与语义事件稳定
- 输出任务相关的语义状态
- 支持老年人摔倒事件识别与稳定触发
- 感知结果能够被任务层直接消费

3. `LTL + 自动机` 的任务规划与多机器人协同
- 任务优先级
- 共享资源互斥
- 电量补给
- 异常恢复与重规划

4. 系统验证
- Gazebo Ignition / TurtleBot4 仿真
- 实车闭环验证
- 空间、语义、任务三个维度的指标化评估

## 2. 当前仓库已经完成的部分

当前仓库已经完成的是“感知层”中的核心子问题：

1. 姿态感知主链路
- `YOLOPose` 人体关键点提取
- 视频 / 摄像头 / 流输入
- GPU 推理验证

2. 跌倒检测原型
- 规则法 baseline
- 学习型时序模型（当前支持 `LSTM` 与 `TCN`）
- 事件稳定化输出

3. 训练与评估工具链
- 序列数据构建
- 统一训练入口
- 批量评估
- 网格调参
- benchmark 归档与工作记录

结论：
- 当前仓库不是最终系统本体，而是最终系统中的“感知子系统原型”
- 这部分是必要的，但不是课题全部内容

## 3. 当前尚未完成的关键部分

相对于开题文档，当前仍缺：

1. 建图层
- `Kalibr` 标定流程说明与数据链路
- `RTAB-Map` 节点、地图更新、导航输入输出定义

2. 规划层
- `LTL` 任务表达
- 自动机或 PlanSys2 映射
- 多机器人任务分配与资源互斥
- 电量补给与恢复机制

3. 系统联调层
- ROS2 节点拓扑
- 感知状态到任务状态的接口
- Gazebo / TurtleBot4 骨架
- 仿真与实物统一评估流程

## 4. 当前最合理的主线

基于当前项目状态，后续工作应分为两条并行主线：

### 4.1 感知主线
目标：把“摔倒检测 + 事件稳定化”做成可供任务层消费的稳定语义模块。

当前重点：
- 保留 `LSTM` 作为综合主模型
- 将 `TCN` 作为低误报对照模型继续优化
- 用统一评估口径维护 benchmark
- 在不更换主 benchmark 的前提下，用补充训练与 external validation 覆盖慢跌倒、家居场景与遮挡盲区

### 4.2 系统主线
目标：把感知层接到开题文档要求的完整机器人任务系统中。

当前重点：
- 已打通 `perception -> supervisor -> planner_request -> planner_status` 最小链路
- 已将 `need_reobserve` 接入系统状态流，并增加最小滞回机制
- 已固定 planner placeholder 的动作、状态、原因码映射
- 当前重点切换为冻结后续 `RTAB-Map / Nav2 / PlanSys2 / LTL` 挂载边界，而不是立即接真实模块

## 5. 当前阶段的工程判断

因此，当前项目的准确定位应表述为：

`面向动态家居场景移动机器人任务系统的感知子系统与整体工程骨架研究框架`

而不是：

`仅用于视频摔倒检测的模型仓库`

## 6. 下一阶段建议

1. 感知侧继续保持 `LSTM` 默认主线，并将额外数据策略收口为补盲区而非换主线
2. 系统侧保持当前冻结接口，不回改 perception / supervisor / planner placeholder 语义
3. 当前 Phase 4a 先按边界接入 TurtleBot4 仿真与 `RTAB-Map` 最小链路
4. 后续再按边界逐步接 `Nav2`、`PlanSys2 / LTL`
5. 用外部验证而不是主 benchmark 替换，评估家居、慢跌倒与遮挡域泛化
6. 逐步从“模型实验仓库”过渡到“系统型课题仓库”
